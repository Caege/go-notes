# Concurrent Web Crawler

A concurrent crawler written in Go.

This project is based on the  [Web Crawler](https://go.dev/tour/concurrency/10) exercise from A Tour of Go, but expanded to support:

- Goroutines
- WaitGroups
- Mutexes
- Shared state protection
- Duplicate URL prevention
- Recursive concurrent crawling
- Simulated web graph generation

---

## Goal

Starting from a root URL, the crawler:

1. Fetches a page
2. Extracts linked URLs
3. Crawls those URLs concurrently
4. Prevents the same URL from being crawled more than once
5. Stops when the maximum depth is reached

---

## Architecture

### Fetcher Interface

```go
type Fetcher interface {
    Fetch(url string) (body string, urls []string, err error)
}
```


The crawler only depends on this interface.

This means the crawler can work with:

* Real HTTP requests
* In-memory data
* Databases
* Files
* Mock implementations

without changing crawler logic.

---

### SafeMap

```go
type SafeMap struct {
    mu sync.Mutex
    m  map[string]Result
}
```

Stores visited pages.

The mutex protects the map because multiple goroutines access it simultaneously.

Without the mutex:

```text
fatal error: concurrent map writes
```

would eventually occur.

---

### Result

```go
type Result struct {
    body string
    urls []string
    err  error
}
```

Stores the crawl result for a URL.

This allows:

* remembering which URLs were visited
* caching fetched content
* inspecting crawl results later

## Duplicate Prevention

Before fetching a URL:

```go
sm.mu.Lock()

if _, ok := sm.m[url]; !ok {
    sm.m[url] = Result{}
} else {
    sm.mu.Unlock()
    return
}

sm.mu.Unlock()
```

A placeholder entry is inserted immediately.

Why?

Imagine two goroutines arrive at the same URL:

```text
Goroutine A -> page-10
Goroutine B -> page-10
```

If neither marks the URL first:

```text
A fetches page-10
B fetches page-10
```

Duplicate work occurs.

The placeholder acts as a reservation:

```text
page-10 -> currently being processed
```

Only one goroutine continues.

## WaitGroup Usage

Each new crawl task increments the WaitGroup:

```go
wg.Add(1)
go Crawl(...)
```

Every crawl completes with:

```go
defer wg.Done()
```

This allows main to wait until every recursive crawl finishes.

```go
wg.Wait()
```

## Fake Web

Instead of performing real network requests, this project generates a random graph.

Example:

```text
page-0
 ├── page-4
 ├── page-12
 └── page-99

page-4
 ├── page-1
 └── page-50
```

Generated by:

```go
GenerateRandomWeb(...)
```

This makes testing deterministic and avoids internet access.

---

## Simulated Network Latency

The fake fetcher includes:

```go
time.Sleep(20 * time.Millisecond)
```

to imitate network delays.

Without it:

* goroutines finish almost instantly
* concurrency is harder to observe

## Concepts Demonstrated

### Concurrency

Creating independent execution paths:

```go
go Crawl(...)
```

---

### Synchronization

Waiting for all work to finish:

```go
sync.WaitGroup
```

---

### Mutual Exclusion

Protecting shared state:

```go
sync.Mutex
```

---

### Recursive Parallelism

Each page can spawn additional crawl jobs.

```text
page-0
 ├── page-1
 │    ├── page-2
 │    └── page-3
 └── page-4
```

Every branch may execute concurrently.

---

### Graph Traversal

The generated website is a graph.

The crawler performs a depth-limited traversal while avoiding revisiting nodes.

## Possible Improvements

### Use RWMutex

Current implementation:

```go
sync.Mutex
```

Could be replaced with:

```go
sync.RWMutex
```

to allow multiple readers simultaneously.

---

### Worker Pool

Current design spawns a goroutine for every discovered URL.

A worker pool could:

* limit memory usage
* limit goroutine count
* provide backpressure

---

### Context Cancellation

Add:

```go
context.Context
```

to support:

* timeouts
* cancellation
* graceful shutdown

---

### Real HTTP Fetching

Replace the fake fetcher with:

```go
http.Client
```

and parse links from actual HTML pages.
